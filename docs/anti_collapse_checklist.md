# AsyncDataLoader 防坍缩约束与验收清单

> 配套文件：与《项目执行手册》《分阶段提示词全集》一起使用。
> 本文件的唯一目的：**保证项目在任何阶段都不会退化成"一个异步文件加载器"**，
> 而是始终保持"有界内存 + 三级重叠 + 顺序可靠落盘"的流水线本质。

---

## 0. 给 AI 的使用说明（每个阶段都贴这一段）

1. 本文件**优先级高于一切"为了省事而简化"的建议**。任何阶段，当你为了让代码更短而想去掉
   背压、有界内存、真实处理 Stage、正确落盘或可观测性时——**不许去掉**，以本文件为准。
2. 每个阶段产出代码后，必须对照下面第 3 节"硬性约束"和第 4 节"验收测试"做一次自检，
   并在阶段交接总结里明确回答第 5 节的自检问题。
3. 如果当前阶段还没到能跑某条验收测试的程度，就**说明它将在哪个阶段被满足**，
   而不是默默绕过。约束可以"还没实现"，但不许"被放弃"。

## 1. 一句话判定标准（贴在心里）

> **文件加载器**回答的是："给我下一块数据。"
> **本项目**回答的是："让读、算、写三件事**同时进行**，内存**永远不超过配置上限**，
> 在**远大于内存**的数据上，产出一个**顺序正确、可靠落盘**的输出文件。"

难点在**协调**，不在"读"。任何让"读"变快但不解决协调问题的实现，都算坍缩。

---

## 2. 什么叫"坍缩"（失败形态，必须避免）

出现以下任何一种，即视为项目已坍缩成简单加载器，必须立即纠正：

- 把整个文件（或整个输出）一次性读进 / 留在内存。
- 把所有数据块读进一个 `vector<Block>` 之后再统一处理。
- 生产者（读）不受消费者（算/写）约束，可以无限超前。
- read → process → write 始终是一个**顺序 for 循环**（它只能作为对照 baseline，不能是最终形态）。
- "处理"阶段只有 NoOp / Checksum 这类占位算子，没有真实 CPU 工作。
- 内存占用随文件变大而变大。
- 把可观测性（metrics）当装饰删掉，导致无法证明上面这些。

## 3. 不可协商的硬性约束（项目的脊梁）

| # | 约束 | 含义 | 必须满足的判据 |
|---|---|---|---|
| H1 | **有界内存 + 背压** | 在飞缓冲区占用的内存有配置上限，与文件大小无关；下游满了，读必须挂起 | 峰值常驻内存 ≈ `block_size × 在飞缓冲区数 + 固定开销`，由配置决定；满则 stall |
| H2 | **三级重叠** | 稳态下 read(i+1)、process(i)、write(i-1) 并发执行 | 端到端耗时 ≈ `N × max(t_read,t_process,t_write) + 填充/排空`，**不是** `N × (t_read+t_process+t_write)` |
| H3 | **流式，不全量载入** | 任何时刻都不要求整文件/整输出在内存 | 文件从 1GB 涨到 50GB、500GB，峰值内存基本不变 |
| H4 | **真实处理 Stage** | process 做真实 CPU 工作（你的预处理算子：滤波/重采样/归一化/校验等） | 换上重处理算子，CPU profile 明显变化，但流水线在重叠下仍受 I/O 限制 |
| H5 | **缓冲区生命周期清晰** | 每块缓冲区在 读→算→写→归还池子 的每一步都有唯一所有者，RAII 归还 | ASan/TSan 下满负荷跑无 use-after-free / data race / double-free / 泄漏 |
| H6 | **输出正确 + 可靠落盘** | 输出与串行 baseline 一致；并行乱序完成也按 offset 写对位置；崩溃不留半成品 | 校验和与 baseline 逐字节一致；写"临时文件 + fsync + rename"原子替换 |
| H7 | **错误与边界不崩** | EOF、短读/短写、文件不存在、backend 不可用、O_DIRECT 对齐错误均被处理 | 对应错误测试全部通过；backend 不可用时自动 fallback |
| H8 | **可观测是受力组件** | 暴露每 Stage 延迟、吞吐、队列深度、在飞缓冲区数 / 内存高水位 | 用 metrics 能指出当前瓶颈在读、算还是写 |

### 内存上限的实现机制（H1/H3 的落地方式）

- 三个配置旋钮：`block_size`、`max_inflight_buffers`（或等价的 `max_memory_bytes`）、`queue_depth`。
- 峰值常驻内存 ≈ `block_size × max_inflight_buffers + 固定开销`，**与文件大小无关**。
- 背压机制：有界队列 + 缓冲区池。池子空了，读阶段挂起（协程 suspend / 线程 wait）；
  消费者归还缓冲区后再唤醒读阶段。**禁止无界队列。**
- 举例（用于满足头号验收 T1）：`block_size = 8MiB`，`max_inflight_buffers = 24` → 约 192MiB，
  留余量给读/算/写各自在飞的块与元数据，总体 ≤ 256MiB，且无论文件是 50GB 还是 500GB 都不变。

## 4. 验收测试清单（这就是"保证不坍缩"的机制）

每条都要可运行、可证伪。**T1 是头号验收，是你明确要的。**

| # | 测试 | 通过标准 | 验证的约束 |
|---|---|---|---|
| **T1** | **有界内存大文件**：`--max-memory=256MB` 处理 **50GB** 文件 | 全程峰值 RSS ≤ ~300MB，不 OOM，正确跑完 | H1,H3 |
| T1b | **规模不变性**：同样 256MB 上限处理 1GB / 50GB / 200GB | 峰值内存基本恒定，不随文件增长 | H3 |
| T2 | **背压消融对照**：提供"无界模式"开关做对照实验 | 无界模式下内存高水位随文件持续增长（用小规模高水位曲线证明即可，不必真跑挂机器）；有界模式恒定 | H1 |
| T3 | **重叠生效**：跑流水线 vs 串行 baseline | metrics 时间线显示读/算/写重叠；端到端 ≈ `max(stage)×N` 而非 `sum×N`；相对串行有明显加速 | H2 |
| T4 | **结果正确**：pipeline 输出 vs 朴素串行 baseline 输出 | 逐字节 / 校验和**完全一致** | H6 |
| T5 | **乱序写正确**：多核并行 process | 输出按 offset 正确，校验和仍与 baseline 一致 | H6 |
| T6 | **崩溃安全**：运行中途 `kill -9` | 最终输出文件要么不存在、要么完整正确；绝不留下半截的"成品"文件 | H6 |
| T7 | **内存/线程安全**：ASan + TSan 满负荷跑 | 无任何报错 | H5 |
| T8 | **fallback**：禁用 io_uring | 自动退化到 threadpool / sync，仍正确跑完 | H7 |
| T9 | **重 CPU Stage**：换上真实重处理算子 | 重叠仍在工作，吞吐仍受 I/O 限制，不退化成串行 | H4,H2 |

> 正确性裁判（oracle）：始终保留"版本 1 串行 baseline"作为 golden output，
> 所有 pipeline 结果都必须与它一致。串行 baseline 不是要被删掉的旧代码，而是测试基准。

## 6. 明确禁止（这些话/做法不许出现）

- ❌ 一次性 `read()` 到一个和文件等大的 buffer；把整文件读进内存。
- ❌ 把所有块攒进 `vector` 再统一处理。
- ❌ 无界队列；生产者不受消费者约束。
- ❌ 顺序 for 循环 read→process→write 当作**最终形态**（只能作为对照 baseline）。
- ❌ 只交付 NoOp / Checksum 作为"处理"的证明。
- ❌ 说"性能来自协程"（协程只负责组织异步代码，不是性能来源）。
- ❌ 编造未实测的性能数字；benchmark 必须如实给出适用边界，
  允许出现"io_uring 在这个顺序负载上没跑赢 threadpool"这类诚实结论。
- ❌ 把 metrics / 可观测性当可有可无的装饰删掉。

---

## 7. 一句话定位（讲项目 / 写简历都用这条）

> 一个**大文件异步预处理流水线**：把几百GB级数据的 read-process-write 从串行改造成
> **三级重叠、有界内存背压、可配置、可观测**的流水线，输出顺序正确且可靠落盘；
> 底层用 io_uring 做异步 I/O、C++20 协程组织异步代码，支持 backend fallback。
> 它**不是**文件加载器——证据是：**256MB 内存上限能正确处理 50GB 文件而不 OOM**。
